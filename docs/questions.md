## 一、核心概念

### 1. 寄存器

| 问题 | 答案 |
| :--- | :--- |
| 寄存器是什么？ | CPU 内部的高速临时存储位置，共 32 个（x0-x31） |
| 能用别名吗？ | 可以！`sp`、`ra`、`a0` 比 `x2`、`x1`、`x10` 更直观 |
| 寄存器不够用怎么办？ | 把暂时不用的数据存回内存（栈上），需要时再读回来 |
| 指令存在寄存器里吗？ | 不！指令存在内存里，CPU 一条条取来执行 |

**寄存器别名速查**：

| 编号 | 别名 | 用途 |
| :--- | :--- | :--- |
| x0 | zero | 永远为 0 |
| x1 | ra | 函数返回地址 |
| x2 | sp | 栈指针 |
| x10-x17 | a0-a7 | 函数参数/返回值 |
| x5-x7, x28-x31 | t0-t6 | 临时寄存器（随便用）|
| x8-x9, x18-x27 | s0-s11 | 保存寄存器（用前要保存）|

### 2. 基础指令

| 指令 | 作用 | 示例 |
| :--- | :--- | :--- |
| `addi` | 加法（带立即数）| `addi x9, x9, 10` |
| `li` | 加载立即数（伪指令）| `li a0, 42` |
| `beq` | 相等时跳转 | `beq a0, a1, label` |
| `and`/`or`/`xor` | 按位与/或/异或 | `and t0, t1, t2` |
| `la` | 加载地址 | `la sp, _stack_top` |
| `sd`/`ld` | 存储/加载双字 | `sd a0, 0(sp)` |
| `call` | 调用函数 | `call kernel_main` |
| `ret` | 返回 | `ret` |

### 3. 汇编器指令（以点开头）

| 指令 | 作用 |
| :--- | :--- |
| `.text` | 代码段开始 |
| `.data` | 数据段开始 |
| `.bss` | 未初始化数据段 |
| `.global` | 声明全局符号 |
| `.section` | 自定义段 |

## 二、内存布局与栈

### 1. 内核内存布局

```
高地址 0x88000000  ┌─────────────────────┐ ← 物理内存上限
                   │                     │
                   │   空闲内存           │
0x80202000  ├─────────────────────┤ ← _stack_top（栈顶，内核最高地址）
                   │   STACK（4KB）      │
0x80201000  ├─────────────────────┤ ← 栈底
                   │   空白缓冲（3KB）    │
0x80200400  ├─────────────────────┤
                   │   .bss/.data/.rodata │
0x80200000  ├─────────────────────┤ ← 内核起始地址（最低地址）
                   │   OpenSBI 固件       │
0x80000000  └─────────────────────┘
```

### 2. 栈的理解

| 问题 | 答案 |
| :--- | :--- |
| 栈是什么？ | 内存中一块 LIFO 区域，存局部变量和函数调用信息 |
| 为什么 C 需要栈？ | 函数调用、局部变量、返回地址都需要栈 |
| 栈的大小？ | 内核 4KB-16KB，你当前用 4KB 足够 |
| 栈的方向？ | 向下增长（从高地址向低地址）|
| sp 是什么？ | 栈指针，指向当前栈顶，是栈的“索引器” |
| `_stack_top` 是什么？ | 栈区域的最高地址边界，编译时固定 |
| sp 和 `_stack_top` 的关系？ | sp 初始指向 `_stack_top`，然后向下移动，永远不会超过它 |
| 栈溢出会怎样？ | 破坏下方的 `.bss`/`.data` 段，导致内核崩溃 |

### 3. 为什么栈不能向上增长？

因为 `_stack_top` 上面不是内核的地盘：
- 可能是空闲内存
- 可能是设备 MMIO 区域
- 可能是未映射内存

**sp 向上移动会跑出栈区域，造成数据损坏或死机。**

## 三、链接脚本（kernel.ld）

| 概念 | 作用 |
| :--- | :--- |
| `OUTPUT_ARCH(riscv)` | 指定输出架构 |
| `ENTRY(_start)` | 指定入口点是 `_start` |
| `. = 0x80200000` | 设置当前位置计数器（内核起始地址）|
| `SECTIONS` | 定义内存布局 |
| `.text : { *(.text) }` | 代码段 |
| `.rodata : { *(.rodata) }` | 只读数据段 |
| `.data : { *(.data) }` | 已初始化数据段 |
| `.bss : { *(.bss) }` | 未初始化数据段 |
| `.stack : { . = . + 4096; _stack_top = .; }` | 栈区域 |
| `PROVIDE` | 条件定义符号 |

**链接顺序**：`.text` → `.rodata` → `.data` → `.bss` → `.stack`（栈最后分配）

## 四、启动流程

```
CPU 上电
    ↓
OpenSBI 固件（M 模式）初始化硬件
    ↓
跳转到 0x80200000（你的内核）
    ↓
start.S: la sp, _stack_top（设置栈指针）
    ↓
start.S: call kernel_main（跳转到 C 入口）
    ↓
main.c: 向 UART 地址写入字符
    ↓
QEMU 显示 "Hello RISC-V Kernel!"
```

### start.S 的作用

- 设置栈指针（sp）
- （可选）清零 BSS 段
- 跳转到 C 语言函数 `kernel_main`

### 为什么必须用汇编？

C 语言运行需要栈，而设置栈本身需要汇编。

## 五、UART（串口输出）

| 问题 | 答案 |
| :--- | :--- |
| UART 是什么？ | 硬件模块，负责串行通信 |
| 地址是多少？ | `0x10000000`（数据寄存器）|
| 如何输出？ | `*(volatile char *)0x10000000 = 'H'` |
| 为什么用 `volatile`？ | 防止编译器优化，确保写入硬件 |
| 为什么不用 `printf`？ | 没有操作系统提供标准库 |
| 谁初始化了 UART？ | OpenSBI 固件 |
| 地址是固定的吗？ | QEMU 中是固定的（`0x10000000`），真实硬件由设备树描述 |
| 每次写入会覆盖吗？ | 不会，UART 内部有 FIFO 队列 |

### UART 的特殊性

```
普通内存：写入 → 存储数据 → 再次写入 → 覆盖
UART：    写入 → 进入 FIFO → 硬件发送 → 自动移除
```

**UART 数据寄存器是“写后即忘”的硬件队列入口，不是普通内存。**

## 六、编译与运行

### Makefile 关键参数

| 参数 | 含义 |
| :--- | :--- |
| `-nostdlib` | 不使用标准 C 库 |
| `-nostartfiles` | 不使用标准启动文件 |
| `-ffreestanding` | 独立环境编译（无操作系统）|
| `-T kernel.ld` | 使用自定义链接脚本 |
| `-c` | 只编译，不链接 |

### QEMU 运行命令

```bash
qemu-system-riscv64 -machine virt -kernel mykernel.elf -nographic
```

| 参数 | 含义 |
| :--- | :--- |
| `-machine virt` | 使用 QEMU 虚拟平台 |
| `-kernel mykernel.elf` | 加载你的内核 |
| `-nographic` | 不弹图形窗口，输出到终端 |

**退出**：`Ctrl+A` 然后 `X`

## 七、常见错误及解决

| 错误 | 原因 | 解决方案 |
| :--- | :--- | :--- |
| `no such instruction: 'la sp,_stack_top'` | 用了 x86 汇编器 | 用 `riscv64-elf-gcc` 编译 |
| `undefined reference to '_stack_top'` | 链接脚本未定义栈顶 | 在 `kernel.ld` 中用 `PROVIDE` 定义 |
| `relocation truncated to fit` | 字符串常量寻址超出范围 | 不用字符串变量，直接输出字符 |
| `warning: RWX permissions` | 段权限不精细 | 可忽略，不影响运行 |
